# Post-hoc analysis & reporting conventions

Conventions for the analysis side of a study: how to turn finished W&B groups into reports,
figures, and JSON summaries that a human or LLM can reproduce, audit, and extend six months
later. Companion to [study-organization.md](study-organization.md) (launch-side provenance)
and [beaker-playbook.md](beaker-playbook.md) (scheduling). The terse rule lives in
[AGENTS.md](../AGENTS.md) §12; this file holds the detail.

This doc is the **target spec for a migration**, not a description of current state.
`FINAL_REPORT.md`, missing `_meta` blocks, and tracked W&B caches in `studies/data-scaling-law/`
are pre-migration artifacts — they will be brought into compliance incrementally.

## Why this exists

- A study generates many runs, many figures, many JSON summaries; without a contract,
  agents and humans silently overwrite each other's outputs and orphan derived files.
- Reports should reproduce **bit-for-bit** on a clean machine from the committed code +
  pinned W&B groups + pinned env. No "you had to be there" knowledge.
- A reader six months later (or a fresh LLM context) must be able to answer, for any
  figure/number: *what W&B groups fed it, what code version produced it, where the source
  lives, how to regenerate it.*
- Provenance lives **inside** the artifacts (frontmatter, `_meta` blocks), not in commit
  messages or external notes — the artifact carries its own audit trail.

## Principles

1. **Reports are code.** Committed, regenerable, diffable. A report file changes only via
   its producer script (modulo manual prose, which is bounded by markers — see below).
2. **One producer per artifact.** Every figure and JSON output is written by exactly one
   script. No silent overwrite. If two scripts produce the same file, one is wrong.
3. **Caches are `.gitignore`'d.** Anything regeneratable from W&B (per-subject pulls,
   intermediate aggregates) is a cache. Reports and curated summary JSON are not caches.
4. **Provenance is in the artifact.** JSON gets `_meta`; reports get YAML frontmatter.
   No reliance on the reader inferring it from file path or recent commits.
5. **Stable boundaries.** Report sections that scripts regenerate are delimited by HTML
   comment markers. Scripts never key off line numbers or fuzzy text matches.
6. **Atomic study unit.** A study's launch configs + analysis scripts + reports stay in
   one folder. Splitting launch from analysis severs provenance.
7. **Idempotent regeneration.** Running `make r<n>` twice in a row produces identical
   files. Order-dependent state is a bug.## Folder layout (per study, target)

```
studies/<study>/
  README.md                # variants index + how to launch
  CHANGELOG.md             # per-study log; one entry per merged PR
  environment.lock         # pinned analysis env (conda list --explicit or pip freeze)
  Makefile                 # one target per report; `all` regenerates everything
  variants/<variant>/
    sweep.yaml
    experiment.yaml
    notes.md
    launch_record_<label>/
      experiment_resumable_submitted.yaml
      sweep.yaml
      results.md           # short post-run stub (see contract below)
  analysis/
    <producer>.py          # one script == one report block (or one JSON producer)
    <producer>.json        # output carrying `_meta`
    fig_<slug>.png         # figure(s) — same producer
    reports/
      r1-<slug>.md         # one file per result, with frontmatter + markers
      r2-<slug>.md
      ...
```

The current `studies/data-scaling-law/analysis/FINAL_REPORT.md` will be split into
`reports/r<n>-<slug>.md` files during migration. `INDEX.md` at `studies/` level lists all
studies; per-study `README.md` lists variants (already covered by §8).

## Report file contract (`reports/r<n>-<slug>.md`)

Header: YAML frontmatter (required keys below). Body: prose + markers around any region a
script will regenerate.

```markdown
---
id: r7
slug: nxd-joint-scaling-grid
status: live              # draft | live | superseded
authors: [han, pck]
wandb_groups:
  - nxd-grid@20260623-102649
  - nxd-grid@20260624-141106
  - v2-sc-active@20260622-144622
inputs:
  script: analysis/nxd_scaling.py
  data: analysis/nxd_scaling.json
  figure: analysis/fig_nxd_scaling.png
reproduce: make -C studies/<study> r7
---

# Result 7 — N x D joint scaling grid

<!-- BEGIN result-7 -->
[regenerated by `analysis/update_reports.py --report r7` — do not edit by hand]

(table, derived numbers, generated bullets)
<!-- END result-7 -->

## Discussion

(free-form prose, edited by humans; outside the markers)
```

Rules:

- Frontmatter is required; missing keys fail the check script.
- `status` transitions: `draft` -> `live` -> `superseded`. `superseded` reports stay in the
  tree (history) but link forward to their replacement in the body.
- Everything between `<!-- BEGIN result-N -->` and `<!-- END result-N -->` is owned by the
  producer script. Manual edits inside the markers will be overwritten without warning.
- Producer scripts MUST match on marker pairs, never on adjacent prose (the kind of
  fragility that deleted Result 8 once already).
- Figures referenced by relative path so the file is portable.

## JSON output contract (`_meta` block)

Every JSON written by an analysis script carries an opening `_meta`:

```json
{
  "_meta": {
    "produced_by": "analysis/nxd_scaling.py",
    "produced_at_pt": "2026-06-24T23:30:00-07:00",
    "dispatcher_git_sha": "<sha>",
    "wandb_groups": ["nxd-grid@20260624-141106", "..."]
  },
  "...": "actual data"
}
```

- Times stamped in `America/Los_Angeles` per AGENTS.md §7.
- The check script verifies `_meta.produced_by` is a real path that exists in the repo.

## `launch_record_<label>/results.md` contract

Written **once** after the launch group settles — a short stub a human can read in 30s:

```markdown
# launch_record_<label>

- W&B group: <variant>@<launch_id>
- W&B project: <project URL>
- Beaker exp: <exp_id> (cluster, priority, gpuCount/memory/cpu)
- Submitted: <Seattle ts>
- Settled: <Seattle ts>
- Status: success / partial (N/M failed) / failed
- Key numbers: <one-line headline metric>
- Feeds reports: r7
- Notes: <anything non-obvious — preemption count, retries, oddities>
```

The check script enforces presence of `results.md` in any `launch_record*/` whose W&B
group has at least one finished run.

## Makefile convention

One PHONY target per report; `all` regenerates everything. Targets call producer scripts
in dependency order:

```makefile
.PHONY: all r1 r7 r8
all: r1 r7 r8

r7:
	python analysis/nxd_scaling.py
	python analysis/rl_baseline.py           # writes overlay onto fig from r7
	python analysis/update_reports.py --report r7

r8:
	python analysis/rl_baseline.py
	python analysis/update_reports.py --report r8
```

If two reports share a producer (e.g. `rl_baseline.py` writes both r7 overlay and r8
core), run it once and let both targets depend on the same stamp. Cross-producer figures
(today: `rl_baseline.py` writes onto `fig_nxd_scaling.png`) violate **one producer per
artifact** and will be flagged by the check script.

## `.gitignore` policy

Cache files (anything pulled from W&B and re-derivable) MUST be `.gitignore`'d:

- `analysis/_cache_*.json`
- `analysis/gru_per_subject.json` and similar per-subject pulls
- `analysis/__pycache__/`

Curated summary JSON (`nxd_scaling.json`, `rl_baseline.json`) and reports/figures are NOT
caches — they are committed.

## Enforcement layers

| Layer | Mechanism | Status |
|---|---|---|
| 1 — Documentation | This file; `AGENTS.md` cross-reference | this PR |
| 2 — Pre-commit + CI | `scripts/check_analysis_artifacts.py` invoked by `pre-commit` and GitHub Actions | not yet |
| 3 — Editor-specific | OpenCode skill / Cursor rule | **skipped for portability** (Layer 2 catches what matters) |

`scripts/check_analysis_artifacts.py` (target) validates:

- Every `studies/*/analysis/reports/r*.md` has valid frontmatter with required keys.
- Every `<!-- BEGIN result-N -->` has a matching `<!-- END result-N -->`.
- Every `studies/*/analysis/*.json` (other than caches) has `_meta` with required keys.
- Every `launch_record*/` with finished W&B runs has a `results.md`.
- `_meta.produced_by` points to a real path.
- No two analysis scripts write the same figure path (grep for `savefig`).
- Cache patterns are present in `.gitignore`.

## Multi-agent collaboration rules

These are the social contracts that make multi-LLM + multi-human work survivable:

- **Never overwrite a `live` report's regenerated region without re-running its producer**;
  hand-editing inside markers loses on the next regen.
- **One PR = one report or one producer change.** A PR that bumps r7 must not silently
  change r1's numbers.
- **If you introduce a new analysis output, declare it.** Add the path to the relevant
  report's `inputs:` block (or create a new report), add `_meta` to JSON, add cache pattern
  to `.gitignore` if applicable.
- **Don't squash-merge** (AGENTS.md §9 — more important here than for code, since each
  regen is a checkpoint of "what the report said at this commit").
- **State assumptions in the PR body** when changing what a report claims; cite the W&B
  group(s) that justify the change.
